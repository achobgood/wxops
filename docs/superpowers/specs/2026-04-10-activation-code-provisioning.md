# Activation Code Provisioning for Firmware-Convertible Devices

**Date:** 2026-04-10
**Status:** Spec
**Scope:** CUCM migration pipeline — device provisioning flow for CONVERTIBLE phones

---

## 1. Problem Statement

The current `handle_device_create` handler in `src/wxcli/migration/execute/handlers.py` (line 388)
uses a single provisioning flow for all devices: MAC-based creation via `POST /devices` with a MAC
address in the body. This works for NATIVE_MPP phones (9800-series, 8800-series MPP, 6800-series)
whose MACs are already known and whose firmware is Webex-ready.

Firmware-convertible phones (8845, 8851, 8865, 7821, 7841, 7861) classified as
`DeviceCompatibilityTier.CONVERTIBLE` follow a fundamentally different registration path:

1. The phone is factory-reset on the CUCM side
2. Enterprise firmware is replaced with MPP firmware via TFTP/EDOS
3. The phone boots into MPP mode with **no cloud registration** — it has no Webex identity yet
4. An **activation code** is entered on the phone's screen (or via QR code)
5. The phone registers to Webex Cloud and associates with its owner (person or workspace)

The MAC-based `POST /devices` flow assumes the device is already Webex-aware and can be
pre-provisioned by MAC. A freshly firmware-converted phone is not Webex-aware — it needs an
activation code to establish its cloud identity. Attempting MAC-based creation for a converted
phone will either fail (phone not recognized) or create a phantom device record with no
actual phone behind it.

### Current Behavior

In the planner (`planner.py` line 231), `_expand_device` generates `device:create` and
`device:configure_settings` operations for all non-skipped, non-WEBEX_APP, non-INFRASTRUCTURE
devices. The handler at line 388 builds a body with `mac` and `model` and POSTs to `/devices`.
There is **no branch** based on `compatibility_tier` — CONVERTIBLE and NATIVE_MPP devices
take the same path.

The `DEVICE_FIRMWARE_CONVERTIBLE` decision (generated by `DeviceMapper` at line 209) has an
"accept" option that says "Convert firmware... then register to Webex via activation code" but
the execution layer does not implement this second half. The decision influences whether the
device is included in the plan (skip = excluded), but when accepted, the handler still does
MAC-based creation.

### Impact

- Firmware-convertible phones that are accepted for conversion will fail to register
- The migration plan promises activation-code-based registration but delivers MAC-based creation
- No mechanism to generate, collect, and distribute activation codes to field technicians
- At scale (100+ convertible phones), this is a migration-blocking gap

## 2. Two Device Provisioning Flows

### 2a. MAC-Based Creation (Existing — NATIVE_MPP)

For phones already running MPP/PhoneOS firmware with known MAC addresses:

```
POST /v1/devices
{
  "mac": "AABBCCDDEEFF",
  "model": "DMS Cisco 8851",
  "personId": "Y2lz..."    // or workspaceId
}
```

Response: `201 Created` with full device record including `id`.

The phone, already Webex-aware, picks up its configuration on next boot/reboot. No user
interaction required beyond plugging in the phone.

**When to use:** `compatibility_tier == NATIVE_MPP` and MAC is known.

### 2b. Activation Code Creation (New — CONVERTIBLE)

For phones undergoing firmware conversion where the MAC alone cannot establish registration:

```
POST /v1/devices/activationCode
{
  "personId": "Y2lz...",         // or workspaceId
  "model": "DMS Cisco 8851"      // required for phones (not RoomOS)
}
```

Response:
```json
{
  "code": "5414011256173816",
  "expiryTime": "2017-11-16T23:38:03.215Z"
}
```

Key differences from MAC-based:
- **No MAC required** in the request body — the phone identifies itself via the code
- **`model` is mandatory** for phones (without it, code only works on RoomOS devices)
- **Returns a 16-digit activation code** that must be entered on the phone
- **Code has an expiry** — must be used before `expiryTime`
- **Either `personId` or `workspaceId`** — not both, same as MAC-based
- **Scope requires** `spark-admin:devices_write` + one of `identity:placeonetimepassword_create`
  or `identity:one_time_password`

**When to use:** `compatibility_tier == CONVERTIBLE` and `DEVICE_FIRMWARE_CONVERTIBLE` decision
resolved as "accept" (convert firmware).

### 2c. Decision-Flow Summary

```
DeviceMapper classifies phone
  |
  +-- NATIVE_MPP ---------> device:create (MAC-based POST /devices)
  |
  +-- CONVERTIBLE --------> DEVICE_FIRMWARE_CONVERTIBLE decision
  |     |
  |     +-- accept -------> device:create_activation_code (POST /devices/activationCode)
  |     +-- manual -------> No device op (replaced with different phone)
  |     +-- skip ---------> No device op (user migrates without device)
  |
  +-- WEBEX_APP ----------> No device op (Webex App replaces soft client)
  |
  +-- INCOMPATIBLE -------> DEVICE_INCOMPATIBLE decision
  |     +-- skip/manual --> No device op
  |
  +-- INFRASTRUCTURE -----> No device op (CUCM infrastructure, not a user device)
```

## 3. Webex Target APIs

### 3a. Create Device Activation Code

**Endpoint:** `POST /v1/devices/activationCode`
**OpenAPI location:** `specs/webex-cloud-calling.json` line 37389
**Scopes:**
- `spark-admin:devices_write`
- Plus one of: `identity:placeonetimepassword_create` (workspaces only) or
  `identity:one_time_password` (workspaces or persons)

**Request body:**
| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `workspaceId` | string | One of workspaceId/personId | Workspace to associate device with |
| `personId` | string | One of workspaceId/personId | Person to associate device with |
| `model` | string | Yes (for phones) | Must match supported device model string, e.g. "DMS Cisco 8851" |
| `orgId` | string | No (query param) | Organization; defaults to caller's org |

**Response (200 OK):**
| Field | Type | Notes |
|-------|------|-------|
| `code` | string | 16-digit activation code |
| `expiryTime` | datetime (ISO 8601) | Code expiration timestamp |

**Error cases:**
| Status | Cause |
|--------|-------|
| 400 | Both workspaceId and personId supplied, or invalid model |
| 403 | Missing required scope (`identity:one_time_password`) |
| 404 | Person/workspace not found |
| 409 | Device limit reached for person/workspace |

**Gotcha:** If `model` is omitted, the activation code will only be accepted on RoomOS devices,
not on phones. For CUCM-converted phones, `model` is mandatory. The model string must come from
the supported devices list (`GET /telephony/config/supportedDevices`).

### 3b. Model String Mapping

The activation code API expects the **Webex model string** (e.g., `"DMS Cisco 8851"`), not the
CUCM model name. The existing `handle_device_create` already passes `data.get("model")` which
the DeviceMapper populates. However, for CONVERTIBLE phones, the DeviceMapper needs to ensure
the model string is in Webex format.

Current model strings from DeviceMapper (`device_mapper.py` line 131):
- Model comes from the raw CUCM phone's `model` field
- For the handler, it's passed through as-is

The `handle_line_key_template_create` handler already remaps CUCM model names to Webex format
for PhoneOS phones. The activation code handler needs the same mapping for CONVERTIBLE models:

| CUCM Model | Webex Model String |
|-----------|-------------------|
| Cisco 8845 | DMS Cisco 8845 |
| Cisco 8851 | DMS Cisco 8851 |
| Cisco 8861 | DMS Cisco 8861 |
| Cisco 8865 | DMS Cisco 8865 |
| Cisco 7821 | DMS Cisco 7821 |
| Cisco 7841 | DMS Cisco 7841 |
| Cisco 7861 | DMS Cisco 7861 |

## 4. Pipeline Integration

### 4a. Planner Changes (`planner.py`)

`_expand_device` (line 231) currently produces the same ops for all non-skipped devices.
It needs to branch based on `compatibility_tier`:

```python
def _expand_device(obj: dict[str, Any]) -> list[MigrationOp]:
    cid = obj["canonical_id"]
    # Existing skip logic for webex_app, infrastructure
    if obj.get("compatibility_tier") in ("webex_app", "infrastructure") or \
       obj.get("model") in _NON_WEBEX_DEVICE_MODELS:
        return []

    name = obj.get("display_name") or obj.get("mac") or cid
    owner_cid = obj.get("owner_canonical_id")
    loc_cid = obj.get("location_canonical_id")
    batch = loc_cid if loc_cid else None

    deps_create: list[str] = []
    if owner_cid:
        deps_create.append(_node_id(owner_cid, "create"))

    tier = obj.get("compatibility_tier")

    if tier == "convertible":
        # Activation code flow: no configure_settings (phone configures itself after registration)
        return [
            _op(cid, "create_activation_code", "device",
                f"Generate activation code for {name}",
                depends_on=deps_create, batch=batch),
        ]
    else:
        # MAC-based flow (native_mpp, or any other tier that wasn't filtered above)
        return [
            _op(cid, "create", "device", f"Create device {name}",
                depends_on=deps_create, batch=batch),
            _op(cid, "configure_settings", "device", f"Configure device {name}",
                depends_on=[_node_id(cid, "create")], batch=batch),
        ]
```

**Key design decision:** Convertible phones do NOT get a `configure_settings` operation.
After activation-code registration, the phone pulls its configuration from Webex Cloud
automatically. Device settings (line keys, softkeys, etc.) are applied via the line key template
and device layout operations that already exist at tier 7. The `configure_settings` handler
currently handles MPP-specific device-level settings (audio codecs, etc.) which are also
cloud-managed post-registration for converted phones.

### 4b. Handler Changes (`handlers.py`)

Add a new handler for the activation code flow:

```python
def handle_device_create_activation_code(data: dict, deps: dict, ctx: dict) -> HandlerResult:
    """Generate activation code for a firmware-convertible device."""
    body: dict[str, Any] = {}

    # Model is required for phones
    model = data.get("model")
    if model:
        # Ensure Webex model format (DMS prefix)
        if not model.startswith("DMS "):
            model = f"DMS {model}"
        body["model"] = model

    # Resolve owner: person or workspace
    owner_cid = data.get("owner_canonical_id")
    if owner_cid:
        owner_wid = deps.get(owner_cid)
        if owner_wid:
            if owner_cid.startswith("user:"):
                body["personId"] = owner_wid
            elif owner_cid.startswith("workspace:"):
                body["workspaceId"] = owner_wid

    return [("POST", _url("/devices/activationCode", ctx), body)]
```

Register in `HANDLER_REGISTRY`:
```python
("device", "create_activation_code"): handle_device_create_activation_code,
```

### 4c. Registration in `__init__.py`

Add to `TIER_ASSIGNMENTS`:
```python
("device", "create_activation_code"): 3,  # Same tier as device:create
```

Add to `API_CALL_ESTIMATES`:
```python
"device:create_activation_code": 1,  # POST /devices/activationCode
```

### 4d. Dependency Graph (`dependency.py`)

The `_CROSS_OBJECT_RULES` for device:create already handle the owner dependency (device depends
on its owner user/workspace being created first). The new `create_activation_code` op gets the
same dependency via the `depends_on` list set by the planner, so no new cross-object rules needed.

However, device_layout:configure and softkey_config:configure operations depend on `device:create`
being completed (they need the Webex device ID). For activation-code devices, the Webex device ID
is not immediately available — the phone hasn't registered yet. This means:

**Design decision:** Device layout and softkey configuration for convertible devices should be
**deferred** until post-registration. Two options:

1. **Option A (recommended):** Skip device_layout:configure and softkey_config:configure for
   convertible devices. Apply line key templates via bulk job after phones register (see the
   bulk operations spec). The activation code output includes enough information for the operator
   to track which phones still need layout configuration.

2. **Option B:** Create a "pending registration" state where the activation code op completes
   successfully (code generated) but downstream layout ops are held in a `waiting_registration`
   status. A separate post-registration sweep picks them up. This adds complexity to the
   runtime state machine.

Recommendation: Option A. The bulk `applyLineKeyTemplate` job is the right tool for post-
registration layout application at scale.

## 5. Activation Code Output and Distribution

### 5a. Capturing Activation Codes

The engine (`engine.py`) already captures the response body for each operation. For activation
code operations, the response contains `code` and `expiryTime`. These need to be:

1. **Stored in the plan_operations table** — the `webex_id` field stores the activation code
   (not a device ID, since the device doesn't exist yet). The `result_body` column captures
   the full response including expiry.

2. **Surfaced in the export** — the deployment plan export (`export/deployment_plan.py`) needs
   a new section: "Activation Codes" that lists each convertible device with its:
   - Device name (CUCM)
   - Owner name and email
   - Model
   - Activation code
   - Expiry timestamp
   - Location

### 5b. Export Formats

**CSV export** — Add `activation_codes.csv` to the export bundle:
```csv
device_name,owner_name,owner_email,model,activation_code,expires_at,location
SEP001122AABBCC,John Smith,jsmith@corp.com,Cisco 8851,5414011256173816,2026-04-20T23:38:03Z,Building A
```

**Deployment plan section** — Add to the markdown deployment plan:
```markdown
## Activation Codes

The following phones require activation codes after firmware conversion.
Distribute codes to the user or on-site IT staff before the conversion window.

| Device | Owner | Model | Code | Expires |
|--------|-------|-------|------|---------|
| SEP001... | John Smith | 8851 | 5414-0112-5617-3816 | 2026-04-20 |
```

**Formatted code display** — Activation codes are 16 digits. Display as groups of 4 for
readability: `5414-0112-5617-3816`.

### 5c. Code Expiry Management

Activation codes have an expiry (the API returns `expiryTime`). If codes expire before phones
are converted, new codes must be generated. The pipeline should:

1. **Log expiry timestamps** in the plan operations table
2. **Warn during preflight** if the planned conversion window extends past the earliest expiry
3. **Support re-generation** — a `wxcli cucm regenerate-codes` command that re-runs activation
   code generation for expired or unused codes, updating the export

## 8. Open Questions

1. **Bulk activation code generation** — The API is per-device (one code per POST). At 500+
   convertible phones, this is 500 sequential API calls. Is there a bulk activation code API?
   Not found in the OpenAPI spec as of 2026-04-10. If not, the per-device approach with
   concurrency (semaphore-bounded, same as current engine) should be adequate — 500 calls at
   20 concurrent = ~25 seconds.

2. **Code expiry duration** — The API docs don't specify the default code lifetime. Empirical
   testing needed to determine whether codes expire in 7 days, 30 days, or some other interval.
   This affects the conversion window planning.

3. **Re-activation on failure** — If a phone fails firmware conversion and needs a new code,
   can the same `POST /devices/activationCode` be called again for the same person/workspace?
   The 409 response suggests there may be a one-code-per-entity limit. Needs verification.

4. **Device settings post-registration** — After a converted phone registers via activation code,
   does it automatically pick up line key template assignments (if the template was already
   created and applied to the location)? Or does a separate `applyLineKeyTemplate` job need to
   run? This determines whether Option A in section 4d is sufficient.
